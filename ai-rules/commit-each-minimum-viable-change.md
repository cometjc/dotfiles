# 每完成一個 Minimum Viable Change 就先提交，並盤點剩餘未提交變更

## 核心要求

- 當這一輪最小可交付變更已落地，且已有對應的最小驗證證據後，應先建立一次本地 commit，再繼續往下做。
- `Minimum Viable Change` 指的是可獨立說明、可獨立驗證、可獨立回滾的最小變更單位。
- 完成第一次 commit 後，應順帶盤點當前仍未提交的變更，嘗試再找出下一個可獨立成立的 Minimum Viable Change。
- 若剩餘變更還無法形成新的 Minimum Viable Change，應明確回報未提交部分與下一個可提交單位之間仍待補完的差距。
- 若使用者明確要求不要 commit、延後 commit，或目前工作樹含有不應代為處理的他人變更，應先遵從限制並改為回報狀態。

## 驗證方式

- 檢查完成一個可獨立驗證的最小變更後，是否有立即建立對應 commit，而不是持續累積在工作樹中。
- 檢查 commit 後是否有主動盤點剩餘未提交變更，並評估能否再拆出下一個 Minimum Viable Change。
- 檢查若無法再拆出下一個 Minimum Viable Change，是否有明確回報未提交部分還缺哪些補完點。
- 檢查使用者明示不要 commit 或工作樹含他人變更時，是否改為回報狀態而非逕自提交。
